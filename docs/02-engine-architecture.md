# Codex Engine — Архитектурный документ

---

## Назначение

Codex Engine — независимый слой логики платформы. Он не знает, как выглядит интерфейс. Он знает, что происходит и что должно произойти дальше.

Player передаёт Engine действие игрока. Engine вычисляет результат и возвращает набор событий. Player реагирует на события визуально.

Studio создаёт данные по схеме Engine. Engine подключает данные без изменения собственной логики.

---

## 1. Подсистемы Engine

### 1.1 Case Engine — Управление делами

Отвечает за жизненный цикл расследований.

Знает, какие дела существуют, в каком они состоянии, какие условия нужны для открытия каждого дела, какой порядок прохождения установлен.

Принимает решение: доступно ли дело конкретному игроку прямо сейчас.

Реагирует на открытие дела, прохождение миссий, закрытие дела.

При закрытии дела передаёт управление в Reward Engine и Progress Engine.

---

### 1.2 Mission Engine — Управление миссиями

Миссия — структурная единица внутри дела. Дело состоит из одной или нескольких миссий.

Mission Engine отвечает за последовательность миссий, условия перехода между ними, ветвление сюжета (если одна миссия ведёт к разным следующим в зависимости от результатов).

Сообщает Case Engine о завершении текущей миссии. Получает от Case Engine разрешение на запуск следующей.

---

### 1.3 Task Engine — Управление задачами

Задача — атомарная единица действия. Внутри каждой миссии одна или несколько задач.

Task Engine управляет состоянием задачи: запущена, выполняется, отправлена на проверку, прошла, провалена.

Отслеживает попытки. Знает лимит попыток для каждой задачи. При исчерпании попыток — отправляет событие в Hint Engine и Event Bus.

Передаёт решение игрока в Evaluation Engine. Получает результат. В зависимости от результата переводит задачу в состояние «пройдена» или «провалена».

---

### 1.4 Evaluation Engine — Проверка решений

Получает решение игрока и эталонный набор тестов задачи.

Запускает решение против каждого теста. Формирует отчёт: какие тесты прошли, какие нет, какой вывод получен, какой ожидался.

Возвращает структурированный результат в Task Engine.

Не знает, что происходит с результатом дальше. Не начисляет награды. Не обновляет прогресс.

Evaluation Engine универсален: одна и та же логика проверки работает для любого языка и любой предметной области, если тесты описаны по единой схеме.

---

### 1.5 Evidence Engine — Управление уликами

Улика — игровой объект, который игрок получает в процессе расследования.

Evidence Engine знает, какие улики существуют в деле, какие условия нужны для открытия каждой улики, в каком порядке они раскрываются.

Условием открытия может быть: завершение задачи, прочтение сообщения, прохождение миссии, получение определённого количества XP.

При выполнении условия Evidence Engine генерирует событие «Улика открыта» и передаёт его в Event Bus.

---

### 1.6 Story Engine — Управление нарративом

Знает сюжетное состояние дела: какие события произошли, какие персонажи задействованы, какова текущая сюжетная точка.

Реагирует на игровые события из Event Bus и определяет, какое нарративное событие должно произойти следующим.

Например: задача пройдена → Story Engine знает, что сейчас нужно показать реакцию куратора → передаёт команду в Dialogue Engine.

Story Engine — это режиссёр. Он не генерирует текст. Он выбирает, какой текст показать и когда.

---

### 1.7 Dialogue Engine — Управление диалогами и сообщениями

Хранит все диалоги, сообщения и реплики персонажей дела.

Получает команду от Story Engine: показать диалог с персонажем X в точке Y.

Формирует событие для Player: «Получено сообщение от персонажа, текст такой-то».

Отслеживает прочитанные сообщения. Знает, какие сообщения зависят от действий игрока (ветвление).

---

### 1.8 Learning Engine — Образовательный слой

Хранит учебную программу: темы, понятия, навыки и их порядок.

Каждая задача в деле помечена скрытым образовательным тегом: какую тему она отрабатывает.

Learning Engine отслеживает, какие темы игрок уже встречал, какие отработал достаточно, какие ещё не затронуты.

При завершении задачи Learning Engine обновляет карту навыков игрока. Player никогда не видит эту карту напрямую. Player получает только игровой эффект: открыт новый инструмент, разблокировано новое дело, пришло сообщение от куратора.

Learning Engine также определяет, когда навык считается освоенным. Это решение принимается по количеству успешных задач с этим тегом, качеству решений, времени между повторениями.

---

### 1.9 Progress Engine — Прогресс и ранг

Отслеживает суммарный прогресс игрока: XP, текущий ранг, серии успехов, общее количество закрытых дел, количество задач по каждой теме.

Принимает события от других подсистем и обновляет показатели.

Вычисляет, достиг ли игрок порога для повышения ранга. При достижении — генерирует событие «Ранг повышен» и передаёт его в Event Bus.

---

### 1.10 Reward Engine — Начисление наград

Получает событие о завершении задачи, миссии или дела.

По правилам дела вычисляет награду: сколько XP начислить, сколько кредитов, какие предметы выдать.

Учитывает модификаторы: первое прохождение, скорость, количество использованных подсказок, количество попыток.

Передаёт результат в Progress Engine и генерирует событие для Player.

---

### 1.11 Hint Engine — Управление подсказками

Знает структуру подсказок каждой задачи. Подсказки уровневые: каждый следующий уровень раскрывает больше, стоит дороже.

Принимает запрос на подсказку. Определяет, какой уровень подсказки доступен игроку. Списывает стоимость. Возвращает подсказку.

Если у игрока исчерпаны попытки — Hint Engine может сам предложить подсказку без запроса.

Если игрок застрял и запросил подсказку максимального уровня — Hint Engine передаёт в Event Bus событие, которое Story Engine использует для реакции куратора.

---

### 1.12 Achievement Engine — Достижения

Хранит список достижений и условия их получения.

Условия могут быть простые (закрыть первое дело) и составные (закрыть три дела подряд без подсказок).

Подписан на Event Bus. При каждом событии проверяет, не выполнено ли условие какого-либо достижения.

При выполнении — генерирует событие «Достижение получено» и передаёт в Reward Engine.

---

### 1.13 Unlock Engine — Управление разблокировками

Знает все объекты системы, которые могут быть заблокированы: дела, инструменты, персонажи, темы в магазине, категории дел.

При каждом значимом событии проверяет, не выполнены ли условия разблокировки какого-либо объекта.

Условия разблокировки могут зависеть от: ранга, закрытых дел, освоенных тем, полученных достижений.

При разблокировке генерирует событие и передаёт его в Event Bus.

---

### 1.14 Event Bus — Шина событий

Центральная магистраль, через которую все подсистемы общаются.

Любая подсистема может публиковать событие. Любая подсистема может подписаться на событие.

Event Bus не содержит бизнес-логики. Он только маршрутизирует события между подписчиками.

Player подписан на Event Bus. Всё, что Engine хочет сообщить Player, проходит через шину.

## 2. Игровые сущности

**Player** — игрок. Имеет ранг, XP, кредиты, инвентарь, карту навыков, историю прохождения.

**Case** — дело (расследование). Содержит миссии, улики, персонажей, образовательные теги, условия открытия, награды. Создаётся в Studio, подключается к Engine через схему данных.

**Mission** — миссия внутри дела. Содержит задачи, сюжетные точки, условия перехода к следующей миссии.

**Task** — задача. Содержит условие, тесты, шаблон решения, подсказки, лимит попыток, образовательный тег, награду.

**Evidence** — улика. Игровой объект с нарративным содержанием. Открывается по условию. Содержит текст, изображение или интерактивный элемент.

**Character** — персонаж. Куратор, аналитик, другие роли. Имеет имя, роль, стиль речи, набор реплик. Не имеет поведения — управляется Story Engine и Dialogue Engine.

**Dialogue** — набор реплик и сообщений персонажа. Привязан к сюжетным точкам.

**Tool** — инструмент в инвентаре игрока. Игровой аналог образовательного понятия или концепции. Например, «Дешифратор строк» — это инструмент, за которым стоит тема «строки и их методы».

**Achievement** — достижение. Имеет название, описание, условие получения, иконку, награду.

**Reward** — описание вознаграждения: тип (XP, кредиты, предмет, разблокировка), количество, условие начисления.

**Inventory** — инвентарь игрока. Список инструментов, расходников и предметов.

**Learning Map** — скрытая карта навыков игрока. Список образовательных тем и степень их освоения. Видна только Engine.

**Curriculum** — учебная программа. Список тем, понятий, последовательность изучения. Создаётся отдельно от дел. Используется Learning Engine как эталон.

## 3. Состояния системы

### Player State

`new` — игрок зарегистрирован, онбординг не пройден

`active` — игрок в игре

`blocked` — игрок превысил лимит попыток, ожидает сброса

`ranked_up` — только что получил новый ранг, ожидает нарративной реакции

### Case State

`locked` — недоступно, условия не выполнены

`available` — доступно, но не начато

`in_progress` — взято в работу

`completed` — все миссии пройдены, дело закрыто

`archived` — доступно для просмотра, награды получены

### Mission State

`locked` — предыдущая миссия не завершена

`available` — предыдущая миссия завершена, эта готова к запуску

`in_progress` — миссия запущена

`completed` — все задачи пройдены

### Task State

`available` — задача разблокирована в рамках миссии

`in_progress` — игрок открыл задачу и работает над ней

`submitted` — решение отправлено на проверку

`passed` — решение прошло все тесты

`failed` — решение не прошло, попытка списана

`exhausted` — попытки исчерпаны, задача на паузе

`rewarded` — награда за задачу начислена

### Learning State (на уровне темы)

`unseen` — тема ещё не встречалась

`introduced` — тема впервые задействована в задаче

`in_progress` — тема встречается повторно, навык отрабатывается

`mastered` — тема освоена по критериям Engine

## 4. Переходы между состояниями

### Case

```
locked
  ↓ (выполнены условия: ранг, закрытые дела)
available
  ↓ (игрок взял дело)
in_progress
  ↓ (завершена последняя миссия)
completed
  ↓ (Reward Engine начислил награды)
archived
```

### Mission

```
locked
  ↓ (предыдущая миссия completed)
available
  ↓ (игрок начал миссию)
in_progress
  ↓ (все задачи миссии — rewarded)
completed
```

### Task

```
available
  ↓ (игрок открыл задачу)
in_progress
  ↓ (игрок отправил решение)
submitted
  ↓ (Evaluation Engine вернул результат)
   ├── passed
   │     ↓
   │   rewarded ──→ (переход к следующей задаче)
   └── failed
         ↓ (если попытки не исчерпаны)
         in_progress
         ↓ (если попытки исчерпаны)
         exhausted ──→ (Hint Engine предлагает подсказку / таймер сброса)
```

### События, поступающие от Player в Engine

| Событие | Описание |
|---|---|
| `player.case_opened` | Игрок открыл карточку дела |
| `player.case_taken` | Игрок взял дело в работу |
| `player.task_opened` | Игрок открыл задачу |
| `player.solution_submitted` | Игрок отправил решение |
| `player.hint_requested` | Игрок запросил подсказку |
| `player.evidence_examined` | Игрок изучил улику |
| `player.message_read` | Игрок прочитал сообщение |
| `player.dialogue_continued` | Игрок перешёл к следующей реплике |

| Событие | Описание |
|---|---|
| `case.unlocked` | Новое дело стало доступным |

| `case.completed` | Дело закрыто, показать итоговый экран |
| `mission.started` | Миссия началась |
| `mission.completed` | Миссия завершена |
| `task.started` | Задача открыта |
| `task.check_passed` | Решение прошло все тесты |
| `task.check_failed` | Решение не прошло, показать ошибки |

| `dialogue.triggered` | Диалоговая сцена с персонажем |

## 6. Действия Engine

Engine выполняет действия как реакцию на события.

**Управление состоянием**
— Перевести задачу в состояние `passed` или `failed`
— Перевести миссию в состояние `completed`
— Перевести дело в состояние `completed` или `archived`
— Заблокировать задачу до сброса таймера

**Начисления**
— Начислить XP по формуле с модификаторами
— Начислить кредиты
— Добавить предмет в инвентарь

**Разблокировки**
— Открыть улику
— Открыть новое дело
— Добавить инструмент в инвентарь игрока
— Разблокировать персонажа или категорию дел

**Нарратив**
— Инициировать диалог с персонажем
— Отправить сообщение от персонажа
— Раскрыть сюжетный элемент

**Образование**
— Обновить карту навыков игрока
— Пометить тему как освоенную
— Определить следующую тему к изучению

**Обратная связь**
— Сформировать детальный отчёт по попытке (какой тест провален, почему)
— Передать подсказку нужного уровня

## 7. Жизненный цикл одного дела

### Фаза 0 — Дело заблокировано

Case Engine проверяет состояние Player: ранг, количество закрытых дел, освоенные темы.

Если условия не выполнены — дело остаётся в состоянии `locked`. Unlock Engine периодически пересчитывает это при каждом значимом событии.

### Фаза 1 — Дело открывается

Условия выполнены. Unlock Engine публикует событие. Case Engine переводит дело в `available`. Player получает событие `case.unlocked`.

Story Engine инициирует первое сообщение от куратора. Dialogue Engine формирует текст. Player получает событие `message.received`.

### Фаза 2 — Игрок берёт дело

Игрок отправляет `player.case_taken`. Case Engine переводит дело в `in_progress`.

Story Engine выбирает точку сюжета «открытие дела» и запускает нарративную сцену. Player получает `case.started` с данными для кинематографического вступления.

Mission Engine активирует первую миссию. Она переходит в `available`.

### Фаза 3 — Прохождение миссий

Для каждой миссии:

Task Engine активирует задачи миссии. Игрок проходит задачи (см. жизненный цикл задачи).

По мере прохождения Evidence Engine отслеживает условия улик. При выполнении — публикует `evidence.unlocked`. Story Engine реагирует и может инициировать диалог.

Когда все задачи миссии в состоянии `rewarded` — Mission Engine переводит миссию в `completed`. Если есть следующая — активирует её.

### Фаза 4 — Завершение дела

Последняя миссия завершена. Case Engine переводит дело в `completed`.

Reward Engine вычисляет итоговую награду за дело. Учитывает: количество попыток, использованные подсказки, время. Публикует `xp.awarded`, `credits.awarded`.

Progress Engine обновляет показатели. Проверяет, достигнут ли порог ранга. Если да — публикует `rank.promoted`.

Achievement Engine проверяет условия всех достижений. Публикует `achievement.granted` по тем, что выполнены.

Learning Engine обновляет карту навыков. Если какая-то тема достигла порога освоения — публикует событие, которое Unlock Engine использует для разблокировки следующего дела или инструмента.

Story Engine инициирует финальный диалог. Dialogue Engine формирует заключительное сообщение куратора.

Case Engine переводит дело в `archived`.

### Фаза 5 — Постдело

Player получает итоговый экран: закрытое дело, полученные улики, начисленные награды, разблокированные объекты. Все данные пришли через события — Player не вычислял ничего сам.

## 8. Жизненный цикл одной задачи

### Открытие

Task Engine переводит задачу в `in_progress`. Player получает `task.started` с данными: условие задачи (в игровой формулировке), шаблон решения, доступные подсказки (заголовки, без содержания), количество попыток.

Learning Engine фиксирует: тема, привязанная к задаче, встречается.

### Работа над задачей

Игрок работает. Engine ничего не делает — это монолог Player.

### Отправка решения

Игрок отправляет `player.solution_submitted` с кодом решения.

Task Engine переводит задачу в `submitted`. Передаёт решение и тесты в Evaluation Engine.

### Проверка

Evaluation Engine запускает решение против каждого теста. Формирует отчёт.

Возвращает результат в Task Engine.

### Результат — Успех

Task Engine переводит задачу в `passed`. Публикует событие в Event Bus.

Reward Engine вычисляет награду с модификаторами:
— Первая попытка: бонус XP
— Без подсказок: бонус кредитов
— Быстрее среднего времени: дополнительный бонус (если замер времени включён в схеме дела)

Публикует `xp.awarded`, `credits.awarded`. Task Engine переводит задачу в `rewarded`.

Learning Engine обновляет карту навыков: тема задачи — плюс один успешный проход.

Evidence Engine проверяет, не открывается ли улика по условию «задача пройдена».

Story Engine выбирает нарративную реакцию. Dialogue Engine формирует реплику куратора. Player получает `message.received`.

### Результат — Неудача

Task Engine переводит задачу обратно в `in_progress`. Уменьшает счётчик попыток.

Player получает `task.check_failed` с детальным отчётом: какой тест провален, ожидаемый и фактический вывод.

Если попытки не исчерпаны — игрок продолжает работу.

### Исчерпание попыток

Task Engine переводит задачу в `exhausted`. Запускает таймер сброса.

Hint Engine получает событие и может сам предложить подсказку. Player получает `hint.suggested`.

Player получает `task.attempts_exhausted` с информацией о времени до сброса и предложением подсказки.

### Запрос подсказки

Игрок отправляет `player.hint_requested` с указанием уровня.

Hint Engine проверяет: есть ли у игрока достаточно кредитов, не была ли эта подсказка уже выдана.

Списывает стоимость. Публикует `hint.delivered`. Player получает содержание подсказки.

Reward Engine фиксирует использование подсказки — при дальнейшем вычислении награды это снизит итоговый бонус.

## 9. Образовательная программа в Engine

### Принцип изоляции

Engine знает образовательную программу полностью. Player не знает её никогда. Player работает только в игровой метафоре.

### Структура учебной программы

Curriculum — это граф тем. У каждой темы есть:
— Название (образовательное, не игровое)
— Уровень сложности
— Предшествующие темы (которые нужно освоить раньше)
— Рекомендуемое количество задач для освоения

Каждая задача в деле помечена тегом: к какой теме она относится. Один тег на задачу. Это скрытое поле, созданное в Studio и не передаваемое в Player.

### Как Learning Engine принимает решения

При каждом прохождении задачи Learning Engine обновляет карту навыков.

Карта навыков — это таблица: тема → счётчик успешных прохождений.

Когда счётчик темы достигает порога, заданного в Curriculum, тема считается освоенной.

Освоение темы — это событие, которое Engine транслирует в игровые следствия:

| Что произошло в Learning Engine | Что получает Player |
|---|---|
| Освоена тема «строки» | «Получен новый инструмент: Дешифратор архивов» |
| Освоена тема «циклы» | «Открыто новое дело: Массовая проверка» |
| Освоена тема «условия» | «Куратор прислал сообщение: Ты готов к сложным делам» |
| Освоена тема «функции» | «Разблокирована категория дел: Аналитические расследования» |

Эти соответствия хранятся в схеме учебной программы. Они создаются в Studio. Engine применяет их как правила.

### Адаптивность

Learning Engine может определить, что игрок слабо проходит задачи конкретной темы (много попыток, много подсказок, долгое время).

В этом случае Engine может рекомендовать вернуться к задачам по этой теме. Это событие транслируется в игровую метафору: «Куратор предлагает провести тренировочные учения».

## 10. Подключение новых дел

### Принцип

Engine не содержит данных конкретных дел. Он содержит только правила работы с делами.

Дело — это пакет данных, созданный в Studio по строгой схеме. Engine подключает пакет и запускает по нему стандартную логику.

### Схема дела

Каждое дело описывает:

**Метаданные** — идентификатор, название, образовательные теги, условия открытия (ранг, список закрытых дел, список освоенных тем)

**Миссии** — упорядоченный список с условиями перехода между ними

**Задачи** — для каждой миссии: условие, тесты (вход → ожидаемый выход), шаблон решения, подсказки по уровням, лимит попыток, награда, образовательный тег

**Улики** — список объектов с условиями открытия

**Персонажи** — список задействованных персонажей и их ролей в деле

**Диалоги** — реплики персонажей, привязанные к сюжетным точкам

**Сюжетные точки** — моменты, в которые Story Engine должен что-то сделать (запустить диалог, показать улику, отправить сообщение)

**Награды** — базовая награда за каждую задачу, миссию и дело целиком

### Что делает Studio

Studio предоставляет интерфейс для создания всех перечисленных элементов. На выходе Studio создаёт пакет данных по схеме Engine.

Engine не нужно обновлять, чтобы появилось новое дело. Достаточно передать Engine новый пакет данных.

### Универсальность

Схема дела не зависит от предмета. Дело по Python, дело по SQL, дело по математике или кибербезопасности описываются по одной и той же схеме. Меняется только содержание задач и тесты.

## 11. Взаимодействие Engine и Player

### Принцип

Engine и Player общаются исключительно через события. Engine не вызывает функции Player. Player не вызывает функции Engine напрямую.

Player отправляет действие. Engine возвращает ноль или несколько событий. Player реагирует на каждое событие самостоятельно.

### Что Player отправляет в Engine

Только действия пользователя. Ничего другого.

Player не передаёт вычисленных результатов. Player не принимает решений о прогрессе. Player не знает, что произойдёт после его действия.

### Что Engine возвращает в Player

Набор событий. Каждое событие содержит:
— Тип события
— Данные, необходимые для отображения
— Ничего лишнего

Player обязан уметь обработать любое событие из списка. Player вправе игнорировать событие, если не знает, что с ним делать (это ошибка реализации, а не логики Engine).

### Пример полного цикла

```
Player → player.solution_submitted { код решения }

Engine → task.check_passed { список пройденных тестов }
Engine → xp.awarded { количество }
Engine → credits.awarded { количество }
Engine → evidence.unlocked { идентификатор улики, содержание }
Engine → message.received { персонаж, текст }
Engine → task.rewarded { итоговая награда с модификаторами }
Engine → case.unlocked { идентификатор нового дела }
```

Player показывает каждое из этих событий последовательно или одновременно — по собственной логике отображения. Engine не управляет тем, как именно Player это покажет.

## Ключевые принципы архитектуры

**Разделение ответственности.** Engine знает логику. Studio знает контент. Player знает отображение. Ни один слой не знает о внутреннем устройстве другого.

**Событийная модель.** Все подсистемы взаимодействуют через события. Ни одна подсистема не вызывает другую напрямую. Это позволяет заменять, расширять и тестировать каждую подсистему отдельно.

**Данные отделены от логики.** Engine содержит правила. Дела, задачи, диалоги, учебная программа — это данные. Новый предмет или новое дело не требуют изменений в Engine.

**Образовательный слой невидим для Player.** Learning Engine работает параллельно с игровым слоем. Его решения транслируются только в игровые события. Игрок учится, не зная, что его учат.

**Универсальность.** Evaluation Engine принимает задачу и тесты — не зависит от языка. Learning Engine принимает тему и счётчики — не зависит от предмета. Mission Engine принимает последовательность задач — не зависит от их содержания. Это означает, что Engine одинаково работает для Python, SQL, алгоритмов, математики и любой другой дисциплины.
