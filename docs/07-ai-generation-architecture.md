# Codex — AI Generation Architecture

---

## Назначение

Как Studio генерирует черновики дел: по умолчанию — честно детерминированный шаблонный генератор; с настроенным ключом — настоящий LLM через российский агрегатор AITUNNEL (доступен без VPN, оплата в рублях — прямой доступ к OpenAI/Anthropic из России заблокирован официально).

---

## 1. Gateway / Provider

`services/content-api/ai/gateway.mjs` экспортирует `generateDraftCase({ topic, existingIds })` и `currentProviderName()`. Gateway не знает деталей конкретного провайдера — только его имя из `AI_PROVIDER` (env, по умолчанию `template`) — это архитектурный шов, ради которого подключение LLM не потребовало ни одной правки в `server.mjs` или Studio.

- **`template`** (по умолчанию, без ключа) — `providers/template-provider.mjs`: набор шаблонов по темам (`if`, `while`, `loop`, `dict`, `multi-return`, `generic`), детерминированно собирает валидный (по `validateCase`) черновик дела на Python с полями `[ЗАПОЛНИТЕ]` вместо творческого текста.
- **`aitunnel`** (настоящий LLM, требует `AITUNNEL_API_KEY`) — `providers/aitunnel-provider.mjs`: вызывает OpenAI-совместимый API AITUNNEL (`https://api.aitunnel.ru/v1/chat/completions`), модель по умолчанию — `deepseek-v4-flash` (дешёвая, задача генерации черновика простая, не требует reasoning-модели). Настраивается `AITUNNEL_MODEL`.

`currentProviderName()` возвращает реальное имя активного провайдера — Studio показывает его прямо (`Черновик готов (провайдер: template|aitunnel)`), никогда не выдаёт шаблонную генерацию за LLM и наоборот.

---

## 2. Почему AITUNNEL, а не OpenAI/Anthropic напрямую

OpenAI и Anthropic официально блокируют доступ (включая API) из России с 2024 года — не вопрос выбора, а недоступность. AITUNNEL — российский агрегатор с оплатой в рублях без VPN, OpenAI-совместимый протокол (`messages`/`model`/`Bearer`-токен) — что позволило написать провайдер без стороннего SDK, тем же `fetch`, что и весь остальной проект.

---

## 3. Надёжность и границы

`aitunnel-provider.mjs` не доверяет ответу модели вслепую:
- `evidence[].id` **всегда** перегенерируется нами (`e1`, `e2`, ...), а не берётся из ответа модели — id вставляется в HTML-атрибут без экранирования (см. `03-content-architecture.md`), доверять здесь внешнему тексту нельзя.
- Ответ модели, обёрнутый в ```` ```json ```` — распознаётся и снимается перед парсингом.
- Любой сбой (нет ключа, сеть, невалидный JSON от модели, HTTP-ошибка API) — явная ошибка с понятным сообщением, а не тихий откат к шаблону: методист должен видеть, что генерация не удалась, а не получить неожиданно шаблонный черновик вместо LLM.
- `playable: false` всегда — черновик от LLM, как и от шаблона, не может попасть игроку без проверки методистом (см. `08-content-production-pipeline.md`).

## 4. Что уже проверено на реальном производственном цикле

`CASE-011` прошёл весь путь: AI-черновик (шаблонный) → правка методистом в Studio → игра и проверка в Player → перенос в постоянный сид `data.js`. Это подтверждает, что конвейер публикации работает независимо от того, какой Provider стоит за Gateway — ровно то, что позволило подключить `aitunnel` без изменений вне `ai/`.
