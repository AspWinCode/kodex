# Codex — Database Architecture

---

## Назначение

Честно зафиксировать: **настоящей базы данных в проекте нет**. Это осознанное архитектурное решение для текущего масштаба, а не недосмотр — документ описывает, что есть вместо БД и когда от этого решения стоит отказаться.

---

## 1. Хранилища данных

| Данные | Формат | Расположение | В git? |
|---|---|---|---|
| Сид игровых дел | JS-модуль (`const CASES = [...]`) | `packages/game-data/data.js` | да — источник истины |
| Правки Studio (версии/рецензия) | JSON-файл | `services/content-api/data/studio-content.json` | нет (`.gitignore`) |
| Журнал игровых событий | JSONL (append-only) | `services/content-api/data/events.jsonl` | нет |
| Прогресс агента | JSON в `localStorage` | браузер игрока | не применимо |

Чтение/запись — напрямую `node:fs/promises` (`readFile`/`writeFile`/`appendFile`) в `server.mjs`, без промежуточного слоя.

---

## 2. Почему не БД

Один процесс, один сервер, данные умещаются в память при чтении, конкурентная запись — редкость (несколько методистов, не тысячи пользователей). PostgreSQL/SQLite добавили бы слой миграций, соединений и ORM-абстракций без реальной пользы на этом масштабе — прямое нарушение KISS (`15-engineering-handbook.md`).

Частичный апдейт стора (`store.cases[id] = payload`, а не перезапись файла целиком с нуля) снижает, но не устраняет риск гонки при двух одновременных записях — см. `08-content-production-pipeline.md`, известные ограничения (нет совместного редактирования).

---

## 3. Когда пересматривать это решение

- Реальное количество методистов, одновременно редактирующих дела, начинает вызывать конфликты записи чаще, чем раз в сессию.
- Требуется история/аналитика по объёму, не помещающемуся в память процесса при чтении (`events.jsonl` читается целиком в `readAllEvents()`).
- Появляется требование к серверной синхронизации прогресса агента между устройствами (сейчас — исключительно `localStorage`).

В любом из этих случаев — SQLite как наименее инвазивный следующий шаг (файл, без отдельного сервера БД), не сразу PostgreSQL.

---

## 4. Резервное копирование

`scripts/backup.sh` архивирует `services/content-api/data/` в `.tar.gz`, ежедневный cron на VPS (`0 3 * * *`), 14 копий хранится. Восстановление проверено вручную (файл удалён → восстановлен из архива → данные совпали) — не гипотетическая процедура.
