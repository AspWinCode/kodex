# Codex — Content Architecture

---

## Назначение

Описывает схему одного «дела» (case) — единицы контента, которую видят и Player, и Studio, и Python Runner. Это не аспирационный документ: схема ниже — то, что реально проверяется `services/content-api/server.mjs` (`validateCase`) и `scripts/check.mjs` (Quality Gate) на каждом сохранении и на каждом запуске гейта.

Источник истины по данным — `packages/game-data/data.js` (сид, версионируется в git) и `services/content-api/data/studio-content.json` (правки Studio, вне git, живут на сервере). `packages/game-data/merge.js` сливает их в единый список дел, который видят Player и Studio.

---

## 1. Схема дела

Обязательные поля (проверяются `validateCase`):

- `id` — латиница, цифры, дефис (`^[a-z0-9-]+$`). Вставляется в HTML-атрибуты Player без экранирования — отсюда жёсткая проверка формата, а не просто «непустое значение».
- `fnName` — имя функции, которую должен написать агент.
- `starter` — стартовый код на Python, который видит игрок на верстаке.
- `evidence` — массив улик, каждая: `{ id, name, tests }`, где `tests` — массив `{ args, expect }`. Минимум одна улика, у каждой — минимум один тест. `id` улики так же ограничен `^[a-z0-9-]+$` (вставляется в `data-ev`).
- `hints` — объект `{ "1": текст, "2": текст, "3": текст }` — три уровня наводок Джарвисмена.
- `briefing` — массив реплик куратора перед делом: `{ curator, text }`.
- `finale` — массив реплик куратора после раскрытия: `{ curator, text }`.
- `versions` — массив версий для отчёта закрытия: `{ text, correct }`, минимум одна с `correct: true`.

Необязательные поля:

- `materials` — массив материалов дела (может быть пустым — см. `case-011`): `{ id, type, title, key, x, y, meta: { source, author }, body: [строка | { code }] }`. `id` материала так же ограничен `^[a-z0-9-]+$` (вставляется в `data-doc`).
- `visual` — маркер типа визуализации результата: `'turtle'` (путь MiniTurtle рисуется SVG) или `'chart'` (спецификация MiniChart рисуется SVG). См. `10-technical-architecture.md`.
- Метаданные картотеки: `num`, `title`, `curator`, `rank`, `difficulty`, `rewardCredits`, `rewardRep`, `anno`, `goal`, `suspects`, `playable`.

## 2. Улики и тесты — контракт с Python Runner

`evidence[].tests[].args` — позиционные аргументы вызова `fnName(*args)`. `expect` сравнивается с результатом через `close_enough()` (`services/python-runner/harness.py`) — толерантно к плавающей точке, рекурсивно по спискам/словарям. Порядок улик и тестов внутри улики значим: авточекер останавливается на первом провалившемся тесте внутри улики (см. `12-api-architecture.md`, `POST /api/run`).

Особый случай — глобальное состояние: харнесс исполняет весь код решения **один раз на попытку** и вызывает `fnName` последовательно для всех тестов всех улик в одном процессе. Это намеренно используется в `CASE-021Б` (счётчик через `global`) — состояние реально сохраняется между уликами.

## 3. Слияние Studio-правок (Publishing)

`merge.js` экспортирует:

- `applyStudioOverrides(casesArr)` — мутирует массив на месте, используется Player. Берёт только **approved**-правки (и легаси-правки без записи `meta` — см. `08-content-production-pipeline.md`).
- `mergedCasesView(seedCases)` — неразрушающий вариант для Studio, видит **все** статусы (draft/in_review/approved/changes_requested), включая собственные черновики.

---

## 4. Инварианты и где они проверяются

| Инвариант | Где проверяется |
|---|---|
| Формат `id` дела/улики/материала | `server.mjs: validateCase()`, `scripts/check.mjs` |
| Минимум одна улика, минимум один тест на улику | оба места выше |
| Наличие hints/briefing/finale/versions, хотя бы одна верная версия | оба места выше |
| Синтаксическая корректность `starter` (Python) | не проверяется статически — только при реальном запуске через Python Runner на верстаке |

Расхождение реализации с этим документом устраняется правкой документа, а не наоборот — раздел ведётся синхронно с `validateCase()`.
