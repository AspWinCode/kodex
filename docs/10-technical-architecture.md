## 2. Технологический стек — намеренно минимальный

- **Без сборки.** Нет bundler'а, нет транспиляции. `<script src="...">` напрямую в браузере.
- **Без фреймворка** (ни React, ни Vue) — весь UI на прямом DOM API + шаблонные строки (`el()`, `esc()` в `apps/player/js/ui.js`).
- **Без ORM/БД.** JSON-файлы (см. `11-database-architecture.md`).
- **Один Node-процесс** обслуживает и статику, и API (dev/prod parity — `python -m http.server` не нужен даже локально).
- **Docker только для исполнения кода агента** — единственное место, где изоляция действительно нужна (произвольный пользовательский Python-код).

Обоснование — Engineering Handbook, раздел KISS (`15-engineering-handbook.md`): сложность добавляется только там, где она снимает реальный риск (изоляция исполнения кода), а не «на будущее».

## 3. MiniTurtle / MiniChart — обход отсутствия дисплея

`turtle` (модуль 5) и `matplotlib` (модуль 9) в обычном Python рисуют в окне/PNG. В headless Docker-контейнере без дисплея это не работает, а сравнение изображения с эталоном не вписывается в модель «результат функции = данные для сравнения». Решение: харнесс подменяет библиотеку собственным объектом с тем же набором вызовов (`forward/left/...` для turtle, `bar/line/hist/...` для chart), который **записывает данные** вместо рисунка. Авточекер сравнивает записанные данные как обычную улику; Player отдельно рисует те же данные SVG на клиенте (`renderTurtlePath`/`renderChartSpec` в `apps/player/js/ui.js`).

## 4. Развёртывание

VPS: nginx (TLS, домен `kodex.tirskix.space`) → `proxy_pass 127.0.0.1:4201` → systemd-юнит `codex-content-api` (`Restart=always`). Обновление: `git pull && systemctl restart codex-content-api` (плюс `docker build` при изменении `harness.py`/`Dockerfile`). Резервное копирование — `scripts/backup.sh`, ежедневный cron.

## 5. Нагрузочное тестирование (проведено на продакшн VPS)

Метод: параллельные запросы через `curl` напрямую на VPS (`127.0.0.1:4201`), с мониторингом `uptime`/`free -m` и здоровья остальных (не относящихся к Codex) сервисов на том же сервере до/во время/после каждого прогона — сервер общий, тест намеренно ограничен, чтобы не задеть чужую нагрузку.

| Эндпоинт | Параллельно | Итог | Наблюдение |
|---|---|---|---|
| `GET /api/health` (в процессе, без Docker) | 200 | 0.44с, 200/200 успех | нагрузка на CPU незаметна — не узкое место |
| `POST /api/run` (спавнит Docker-контейнер на попытку) | 10 | 0.64с, 10/10 успех | — |
| `POST /api/run` | 30 | 1.82с, 30/30 успех | — |
| `POST /api/run` | 50 | 3.41с, 50/50 успех | load average кратковременно 8.62 (насыщение всех 8 ядер VPS), восстановление за ~20с |

Единственное реальное узкое место — **спавн Docker-контейнера на каждую попытку проверки** (`services/content-api/runner/python-runner.mjs`), не сам Node-процесс `content-api` (тот легко держит сотни параллельных запросов к «дешёвым» эндпоинтам). Ни разу не наблюдалось ни одной функциональной ошибки вплоть до 50 одновременных проверок; тест остановлен на этой отметке намеренно (общий VPS с полутора десятком чужих проектов), а не потому, что была найдена точка отказа.

Практический вывод: текущее развёртывание на одном VPS выдерживает всплески как минимум в 50 одновременных проверок без сбоев, но устойчивая нагрузка на этом уровне полностью займёт все 8 ядер сервера. Если реальное число одновременных агентов на проверке регулярно приблизится к этой величине — рассмотреть либо более мощный сервер, либо очередь/ограничение частоты перед `/api/run`, сглаживающее всплески.

## 6. Известные ограничения

- Прогресс агента хранится только в `localStorage` браузера — нет серверной синхронизации между устройствами.
- Абсолютный предел (точка отказа) не найден — тест намеренно остановлен на 50 параллельных из соображений совместного использования VPS, см. раздел 5.
