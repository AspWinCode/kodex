# Codex — Content Production Pipeline

---

## Назначение

Полный путь дела от идеи до постоянного контента, играбельного всеми — включая цикл версий и рецензии, добавленный поверх v1.0 (см. `CHANGELOG.md`).

---

## 1. Стадии

```
Идея / AI-черновик (07-ai-generation-architecture.md)
        │
        ▼
Редактирование в Studio (apps/studio) ── PUT /api/content/:id
        │  каждое сохранение = новая версия в history, статус → draft
        ▼
Отправка на проверку ── POST /api/content/:id/submit
        │  статус → in_review
        ▼
Рецензия ── POST /api/content/:id/review { decision, comment }
        │
        ├─ approved ──────────► видно Player (GET /api/content без ?all=1)
        │
        └─ changes_requested ─► обратно на редактирование (статус видим
                                  в Studio с комментарием рецензента)
        │
        ▼
Игра и проверка в Player (реальный геймплей, реальный Python Runner)
        │
        ▼
Перенос в постоянный сид ── ручное копирование в packages/game-data/data.js
                              (коммит в git — источник истины навсегда)
```

---

## 2. Идентификация автора/рецензента

Заголовок `X-Studio-Author`, значение берётся из `localStorage.getItem('studio-author')` браузера (панель в правом верхнем углу Studio). **Не аутентификация** — подделать тривиально, любой может ввести любое имя. Это подпись «на доверии» для одной небольшой доверенной команды, задокументированная как таковая в коде (`merge.js`, `server.mjs`) и здесь. Значение кодируется `encodeURIComponent` на клиенте и раскодируется на сервере — иначе кириллица ломает `fetch()` (HTTP-заголовки ограничены ISO-8859-1).

---

## 3. Видимость: Studio vs Player

`GET /api/content` без параметров — только `approved`-правки **и** легаси-правки без записи `meta` (сохранённые до внедрения цикла рецензии — не гасим уже опубликованный контент задним числом). `GET /api/content?all=1` — полный стор со всеми статусами, используется только Studio (`mergedCasesView`, `isStudioOverridden`).

---

## 4. История версий

Каждое сохранение (`PUT /api/content/:id`) добавляет запись `{ version, savedAt, author, snapshot }` в `history[id]` (капается на последние 20 версий). `POST /api/content/:id/restore { version }` копирует старый снапшот обратно в `cases[id]`, добавляет **новую** версию поверх истории (не удаляет промежуточные — восстановление ведёт себя как «откат вперёд», а не перезапись прошлого) и сбрасывает статус в `draft` (требует повторной рецензии).

---

## 5. Известные ограничения

- Нет совместного редактирования одного дела несколькими людьми одновременно — конкурентные правки одного `id` просто перезаписывают друг друга (последняя запись побеждает).
- Нет уведомлений методисту о решении рецензента — комментарий виден только при следующем открытии дела в Studio.

---

## 6. Пройденный полный цикл (проверено, не гипотетически)

`CASE-011` — первое дело, реально прошедшее весь конвейер до внедрения цикла рецензии. После внедрения рецензии (эта сессия) цикл draft→submit→approve→restore проверен вручную через реальный HTTP API и сквозным прогоном в браузере на `case-001` (тестовая правка, откачена после проверки).
