# Codex — журнал релизов

## LMS-интеграция: обратный канал прогресса (Codex → learning-portal)

Программисты `learning-portal` реализовали приёмный эндпоинт по нашей
спецификации (`docs/17-lms-integration.md`, ранее раздел 3, передана им
файлом `KODEX_PROGRESS_SYNC_SPEC.md`): `POST /api/v1/student-portal/progress-sync`
(таблица `student_course_progress`, подпись `X-Kodex-Signature` тем же
секретом, что и вход) — реальный путь оказался с префиксом `/api/v1`,
не `/api`, учтено при подключении.

Со стороны Codex (`services/content-api/server.mjs`):
- `loadGameData()` / `buildLmsProgressSummary()` — считают сводку прогресса
  (раскрыто дел, ранг, число значков, имя последнего значка) из состояния
  Player и сида дел.
- `pushLmsProgressSummary()` — подписывает сводку HMAC-SHA256 и best-effort
  шлёт на `LMS_PROGRESS_SYNC_URL`, вызывается из `PUT /api/lms-progress/:externalRef`
  сразу после сохранения, не блокируя ответ клиенту.

Проверено: локально (мок-приёмник — тело и подпись совпали побайтово) и
на реальном продакшн-эндпоинте LMS (`https://tirskix.space/api/v1/student-portal/progress-sync`) —
корректные `401`/`404` на невалидные запросы, подтверждающие совместимость
подписи и формата с их стороной.

Это полностью закрывает `docs/16-product-delivery-roadmap.md` — единственный
оставшийся пункт был именно этот обратный канал.

## LMS-интеграция: вход по SSO из learning-portal + серверный прогресс

Изучен реальный репозиторий `learning-portal` (кабинет ученика/родителя,
куда встраивается Codex) — там уже был подготовлен модуль
`app/services/kodex_sso.py` и редирект на `https://kodex.tirskix.space/api/auth/sso`,
ждавший ответной реализации с нашей стороны.

**Реализовано (Codex → готово полностью):**
- `services/content-api/sso/kodex-sso.mjs` — проверка одноразового JWT
  (HS256, TTL 60с, подписан `SSO_KODEX_SHARED_SECRET` — отдельным секретом,
  не основным ни одной из систем), без сторонних библиотек (нативный
  `node:crypto`). Защита от повторного использования токена по `jti`.
- `GET /api/auth/sso` — проверяет токен, редиректит в Player с
  `external_ref`/именем во фрагменте URL (не в query — не логируется).
- `GET/PUT /api/lms-progress/:externalRef` — серверное досье агента,
  ключ — устойчивый `external_ref` из LMS, а не браузер: прогресс
  переживает смену устройства при входе через тот же кабинет.
- `apps/player/js/app.js` (`lmsBootstrap`) заменил прежний прототипный
  SSO-код (ожидавший несуществующий формат `access_token`/`refresh_token`)
  на реальный контракт; `state.js` best-effort пушит состояние на сервер
  при каждом `save()`, если известен `external_ref`.

Проверено сквозным прогоном: реальный JWT (сгенерирован `python-jose`,
как это делает `learning-portal`) → редирект → прогресс сохраняется →
очистка `localStorage` (симуляция другого устройства) → повторный вход
с тем же `external_ref` → прогресс восстановлен с сервера.

**Не реализовано (ждёт программистов `learning-portal`):** обратный канал
(прогресс/значки → LMS, для родительского дашборда) — в `learning-portal`
нет ни таблицы, ни эндпоинта приёма. Полная спецификация — новый
документ `docs/17-lms-integration.md`, раздел 3 (схема таблицы, эндпоинт,
подпись запроса тем же общим секретом, что и вход).

## 10 новых значков за стиль игры (мастерство/настойчивость/объём/исследование)

Поверх модульных значков добавлены 10 значков, не привязанных к конкретной
теме программы, а к тому, **как** играет агент:

- **Мастерство**: «Без подсказок» (раскрыть дело без единой наводки),
  «Высшая сложность» (сложность III, с первой попытки, без наводок).
- **Настойчивость**: «Не сдаюсь» (раскрыть дело после 3+ провалов подряд
  на нём же), «Второе дыхание» (раскрыть дело, верстак которого хотя бы
  раз «остывал»). Специально поощряют возвращение после трудностей, а не
  только идеальное прохождение.
- **Объём**: «Ветеран» (10 дел), «Легенда агентства» (25 дел).
- **Исследование**: «Разведчик полигона» (все учения Полигона),
  «Любопытный агент» (наводка хотя бы раз — не стыдит за помощь).
- **Снабжение/ранг**: «Хорошо укомплектован» (всё снаряжение куплено),
  «Высший допуск» (ранг Спецагент).

Потребовалось расширить `cs` двумя полями: `maxFailStreak` (пик серии
провалов — фиксируется до сброса `failStreak`, иначе к моменту проверки
достижений он уже обнулён) и `hitCooldown` (флаг «кулдаун случался хотя
бы раз»). Значки, не привязанные к раскрытию дела (наводка, покупка,
учение), проверяются напрямую в точке действия
(`overlays.js`/`screens.js`), а не через `checkAchievements()`.

Все 10 проверены реальными сценариями в браузере (включая настоящий клик
по кнопке наводки Джарвисмена для «Любопытного агента»).

## Верстак: подсветка синтаксиса Python (CodeMirror 5)

Редактор кода на верстаке и в Полигоне был голым `<textarea>` — без
подсветки, автоотступов, номеров строк; для Python, где отступ значим,
это была реальная эргономическая проблема, не косметика. Подключён
CodeMirror 5, вендорится локально в `apps/player/vendor/codemirror`
(188 КБ) — не внешний CDN, чтобы верстак не зависел от стороннего
аптайма (Engineering Handbook, раздел 1 — KISS без лишних сетевых точек
отказа для критичной части платформы).

`mountCodeEditor()` (`apps/player/js/ui.js`) — общая обёртка для верстака
и Полигона: подсветка Python, автоотступ на 4 пробела после `:`, Tab как
4 пробела (не перевод фокуса), подсветка парных скобок, номера строк.
Тема `cm-s-kodex` (`apps/player/css/app.css`) — под палитру проекта
(неоново-зелёный/циан), не generic-тема CodeMirror.

Проверено в браузере: автоотступ после `:` (`newlineAndIndent`),
кастомный Tab, подсветка синтаксиса на реальном деле (CASE-017), полный
цикл решения через новый редактор на Полигоне (написание → прогон →
реальный Python Runner → зачтено).

## Настоящий LLM-провайдер (AITUNNEL) — AI Generation больше не только шаблон

Добавлен `services/content-api/ai/providers/aitunnel-provider.mjs` —
второй провайдер Gateway, вызывающий настоящую LLM через AITUNNEL
(российский OpenAI-совместимый агрегатор, оплата в рублях без VPN —
прямой доступ к OpenAI/Anthropic API из России заблокирован официально).
Активируется `AI_PROVIDER=aitunnel` + `AITUNNEL_API_KEY`; без ключа
платформа продолжает честно работать на прежнем шаблонном провайдере —
ничего не меняется молча. Модель по умолчанию — `deepseek-v4-flash`
(дешёвая, достаточная для структурной генерации черновика).

Попутно исправлен реальный баг: `template-provider.mjs` до сих пор
генерировал `starter` в JS-синтаксисе (`function ... {}`), хотя платформа
перешла на Python ещё на этапе Python Runner — приведён к Python-заготовкам.

Ответ модели не считается доверенным: `evidence[].id` всегда
перегенерируется сервером, а не берётся из текста модели (вставляется в
HTML-атрибут без экранирования); markdown-обёртка ответа снимается перед
парсингом; любой сбой — явная ошибка методисту, а не тихий откат к шаблону.
Черновик от LLM получает `playable: false` и проходит тот же гейт качества
и цикл рецензии, что и любая ручная правка — без привилегированного пути.

Это закрывает последний открытый пункт `docs/16-product-delivery-roadmap.md`.

## Studio: совместное редактирование — оптимистичная блокировка версией

Конкурентная правка одного дела двумя методистами больше не перезаписывает
одну правку другой молча. Вместо real-time инфраструктуры (WebSocket/presence)
переиспользована уже существующая история версий: Studio при открытии формы
запоминает номер последней версии (`baseVersion`), при сохранении отправляет
его заголовком `X-Base-Version`; сервер отклоняет запрос `409`, если кто-то
уже сохранил дело позже открытия формы. Methodist видит явный выбор —
обновить форму (потерять свою правку) или сохранить принудительно поверх.

Проверено реальными конкурентными HTTP-запросами (409 на устаревшей версии,
успех на актуальной) и сквозным прогоном в браузере (два «методиста»,
конфликт корректно показан с именем автора и номером версии, принудительное
сохранение создаёт новую версию поверх конкурентной без потери истории).

Это закрывает последнее известное ограничение документа 09 (Studio
Architecture) — см. `docs/00-index.md`.

## Модуль 9 «Визуализация» (matplotlib) — MiniChart + все 9 модулей закрыты

Тот же архитектурный приём, что и для turtle (модуль 5): настоящий
matplotlib рисует в окне или сохраняет PNG — ни то, ни другое не сравнить
с ожидаемым значением, а рендер PNG в контейнере без дисплея ради того же
результата — лишняя сложность. `services/python-runner/harness.py`
добавляет **MiniChart** — объект с методами `bar/line/hist/title/xlabel/
ylabel`, который вместо рисунка записывает спецификацию графика (тип,
подписи, данные) как обычные данные (`new_chart()` в неймспейсе решения).
Авточекер сравнивает спецификацию с эталоном; Player (`renderChartSpec` в
`apps/player/js/ui.js`) рисует ту же спецификацию SVG — столбчатую
диаграмму, линейный график или гистограмму.

Три новых дела с `visual: 'chart'`: CASE-038 (столбчатая диаграмма),
CASE-039 (линейный график), CASE-040 (гистограмма — деление на бины,
подсчёт частот). Проверено сквозным прогоном в браузере со скриншотами:
диаграмма нагрузки по кураторам и гистограмма распределения отрисованы
корректно из реального результата решения.

**39 играбельных дел — все 9 модулей учебной программы теперь закрыты.**

## Модуль 5 «Графика» (turtle) — MiniTurtle + визуализация пути

Настоящий модуль `turtle` рисует в окне (tkinter) — в изолированном
headless-контейнере без дисплея он не запускается, а рисунок нельзя
сравнить с ожидаемым значением так же просто, как остальные улики. Решение:
`services/python-runner/harness.py` внедряет в решение агента функцию
`new_turtle()` — собственный MiniTurtle с тем же набором команд
(forward/left/right/penup/pendown/goto/...), который вместо рисования на
экране записывает пройденный путь как обычные данные (список отрезков).
Авточекер сравнивает этот путь с эталоном как с любой другой уликой; Player
(`apps/player/js/ui.js`, `renderTurtlePath`) отдельно рисует тот же путь как
SVG на экране проверки — учащийся видит настоящий рисунок.

Харнесс дополнительно всегда возвращает `lastResult` (не только при провале,
как раньше `got`) — общее улучшение протокола, не только для turtle.

Три новых дела с полем `visual: 'turtle'`: CASE-022 (квадрат циклом),
CASE-023 (правильный многоугольник по формуле угла 360/n), CASE-024
(пунктирная линия через penup/pendown). Все три проверены сквозным прогоном
в браузере — путь реально совпадает с решением и корректно рисуется (квадрат
и равносторонний треугольник подтверждены визуально).

23 играбельных дела всего.

## Закрыты пробелы в модулях 3–4 (реальный учебный план) — 3 новых дела

20 играбельных дел всего. Модуль 5 (графика, turtle) требует отдельного
архитектурного решения — обычный `turtle` не работает в headless Docker без
дисплея, а его график нельзя проверить простым сравнением возвращаемого
значения; отложено до отдельного обсуждения подхода.

## Поверх v1.0.0 — Python Runner, тех. долг по id, версии и рецензия в Studio

- **Python Runner** — платформа теперь учит и проверяет Python (не JavaScript). Исполнение решения перенесено с браузерного `new Function()` на серверный изолированный Docker-раннер (`services/python-runner`, харнесс на Python с толерантным сравнением чисел и SIGALRM-таймаутом). Все 11 дел v1.0 и 5 упражнений полигона переведены на Python-синтаксис; добавлены два новых дела (numpy — среднее по серии, pandas — поиск максимума в таблице), явно проверяющие, что обе библиотеки реально работают. Проверено сквозным прогоном через браузер и напрямую в продакшн-контейнере на VPS (реальная изоляция `--network=none --read-only`, реальный таймаут на Linux).
- **Тех. долг закрыт**: `evidence[].id` и `materials[].id` теперь проходят ту же проверку допустимых символов, что `id` дела (были тем же классом уязвимости — вставляются в HTML-атрибуты без экранирования).
- **Studio: версии и рецензия** (Studio Architecture, `docs/09`, частично) — каждое сохранение дела в Studio создаёт новую версию в истории (просмотр, восстановление любой прошлой версии). Введён статусный цикл `draft → in_review → approved / changes_requested`: Player теперь видит только одобренные правки (и легаси-правки, сохранённые до этой функции — не гасим уже опубликованное задним числом), а не любое сохранение в Studio вживую, как раньше. Идентификация автора/рецензента — по имени в localStorage браузера (`X-Studio-Author`), явно НЕ аутентификация: подделать тривиально, это подпись «на доверии» для одной команды. Совместное редактирование одного дела несколькими людьми одновременно (конфликты правок) — по-прежнему не реализовано.

## v1.0.0 — первый стабильный релиз

**Важная оговорка о масштабе.** Это не полный образовательный год (на момент релиза оценивался как «8 сезонов, ~66 дел» — оценка была приблизительной и впоследствии уточнена, см. `docs/16-product-delivery-roadmap.md`, реальная программа — 9 модулей) — это честный, полностью проверенный релиз текущего среза платформы: 11 дел, покрывающих модули 1–2 (основы, условия, циклы) и часть модуля 3 (списки, словари, множественные значения). Полный охват программы, версии/рецензия в Studio, настоящий LLM-провайдер вместо шаблонного — следующий этап работы, не часть этого релиза (все три пункта, кроме LLM-провайдера, закрыты в последующих этапах — см. записи выше).

### Что реально работает и проверено

- **Codex Player** — полный игровой цикл (онбординг → хаб → картотека → бриф → материалы → верстак → авточекер → отчёт → награды → ранги) на 11 играбельных делах.
- **Codex Studio** — список дел, редактирование, гейт качества на сервере, экспорт правок, дашборд аналитики.
- **Реальный Publishing** — правки Studio хранятся на сервере (`services/content-api`), видны Player с любого браузера/устройства, не в localStorage.
- **AI Generation** — Gateway/Provider абстракция; провайдер по умолчанию честно детерминированный (шаблонный), не выдаётся за настоящую LLM.
- **Analytics** — реальный журнал прохождений (не демо-числа), агрегаты по каждому делу в Studio.
- **Один полный производственный цикл** — дело `case-011` прошло путь AI-черновик → правка в Studio → игра и проверка в Player → перенос в постоянный сид.
- **Security review** — найдены и исправлены: неэкранированное название дела на экране рангов, отсутствие проверки допустимых символов в `id` дела (оба — реальные stored-XSS-векторы через контент, а не гипотетические).
- **Резервное копирование** — `scripts/backup.sh`, ежедневный cron на VPS, восстановление проверено вручную (файл удалён → восстановлен из архива → данные совпали).
- **Отказоустойчивость** — процесс `content-api` убит намеренно (`kill -9`) → `systemd` (`Restart=always`) поднял его заново за ≤4 секунды без вмешательства.
- **Полная регрессия** — все 11 играбельных дел пройдены эталонными решениями в одном прогоне, все улики подтверждены у каждого.
- Развёрнуто на `kodex.tirskix.space` (VPS, nginx → systemd-сервис `codex-content-api`).

### Известные ограничения (осознанно не входят в этот релиз)

- `id` вложенных объектов (`evidence[].id`, `materials[].id`) не проходят ту же проверку допустимых символов, что `id` дела — остаточный риск для доверенного внутреннего инструмента (см. README, раздел «Безопасность»).
- Studio: нет версий/истории изменений, нет рецензии, нет совместного редактирования одного дела несколькими людьми (Studio Architecture, `docs/09`, — реализована только первая, минимальная часть).
- AI Generation: нет реального LLM-провайдера (нет ключа доступа) — по умолчанию генерируется валидный, но текстово-шаблонный черновик.
- Нагрузочное тестирование не проводилось — платформа не проверялась на реальных объёмах одновременных пользователей.
- Документы 03–16 (`docs/`) согласованы как архитектурная спецификация, но не все перенесены в файлы репозитория дословно (см. `docs/00-index.md`).
