# Kodex / Codex

Образовательная игровая платформа: обучение программированию под маской работы в детективном агентстве «Кодэкс».

## Структура репозитория

```
kodex/
├── apps/
│   ├── player/            — Codex Player: игровой клиент агента
│   └── studio/              — Codex Studio: минимальный редактор дел для методистов
├── docs/                    — архитектурная документация платформы (источник истины)
├── packages/
│   └── game-data/            — общие данные мира (дела, кураторы, ранги) + клиент content-api,
│                                читается и Player, и Studio
├── services/
│   └── content-api/           — единственный сервис проекта: раздаёт статику + хранит
│                                 Studio-правки в общем JSON-файле (реальный Publishing, без БД)
├── scripts/                   — служебные скрипты (Quality Gate для CI)
└── .github/workflows/          — CI
```

## Запуск локально

```bash
node services/content-api/server.mjs
```

Один процесс на весь проект — раздаёт статику (Player, Studio, общие пакеты) и обслуживает API правок. Никакой сборки, никаких зависимостей вне встроенных модулей Node.

- Codex Player: `http://localhost:4173/apps/player/` (или просто `http://localhost:4173/` — корневой `index.html` редиректит)
- Codex Studio: `http://localhost:4173/apps/studio/`

Player и Studio обязаны работать на одном origin — они делят не localStorage (как было в v0.3), а общий сервис `content-api`: правки, сделанные в Studio, хранятся на сервере в `services/content-api/data/studio-content.json` (вне git — см. `.gitignore`) и видны Player с любого браузера и устройства.

### Кэш при локальной разработке

Скрипты и стили подключены с версией (`?v=3`) для обхода браузерного кэша. При следующем ощутимом изменении бампните версию в соответствующем `index.html` или используйте жёсткую перезагрузку (Ctrl+Shift+R).

## Codex Player

Одностраничное приложение без сборки — статические `HTML/CSS/JS`.

| Файл | Назначение |
|---|---|
| `apps/player/js/app.js` | Каркас, SSO-bootstrap, маршрутизация (ждёт готовности контента перед первым рендером) |
| `apps/player/js/state.js` | Состояние агента, персистентность (localStorage), мутации прогресса |
| `apps/player/js/engine.js` | Формализация Engine Architecture поверх state.js: именованные состояния Case/Task, журнал игровых событий (`S.events`) |
| `apps/player/js/content-overrides.js` | Асинхронно накладывает Studio-правки (через content-api) на сид `CASES` при загрузке |
| `apps/player/js/screens.js` | Рендер экранов (хаб, картотека, кабинет расследования, онбординг) |
| `apps/player/js/overlays.js` | Сквозные слои: канал связи, Джарвисмен |
| `apps/player/js/ui.js` | Примитивы интерфейса, авточекер решений |

### Вход

Регистрации и авторизации внутри приложения нет — вход выполняется через SSO (токены во фрагменте URL: `#access_token=...&refresh_token=...&user=<base64url>`). Без параметров SSO в проде — редирект на внешний портал; в текущем прототипе создаётся сессия напрямую для локальной разработки.

## Codex Studio

Первая версия Studio Architecture (`docs/09`) — список дел + форма редактирования одного дела поверх тех же данных, что читает Player.

- Список дел с пометкой статуса: **сид** (из `packages/game-data/data.js`, не тронуто), **изменено** (Studio-правка поверх сида), **новое** (создано в Studio, в сиде не существует).
- Форма редактирования: простые поля структурированы (название, куратор, награды, стартовый код и т.д.), сложные вложенные (улики/тесты, подсказки, брифинг, версии отчёта, финал) — как JSON-текст.
- **Гейт качества при сохранении выполняет сервер** (`services/content-api/server.mjs`, та же логика инвариантов, что в `scripts/check.mjs`): дело не сохранится без функции, тестов у каждой улики, подсказок, брифинга, финала и хотя бы одной верной версии отчёта. Ответ 422 со списком нарушений возвращается прямо в форму.
- «Сбросить» — удаляет Studio-правку на сервере, дело возвращается к сиду.
- «Экспорт правок» — показывает JSON всех Studio-правок для ручного переноса в `packages/game-data/data.js`, чтобы зафиксировать их как новый сид (следующий шаг Publishing, см. `docs/08`).
- «✨ Сгенерировать черновик» — запрашивает у AI Gateway готовый скелет дела под выбранную тему (условие, цикл, словарь и т.д.), открывает его в форме редактирования для доработки. Черновик всегда создаётся с `playable: false` — методист обязан осмысленно включить дело после правки текстов `[ЗАПОЛНИТЕ]`.

## AI Generation (v0.4)

`services/content-api/ai/` — генерация черновика дела по кнопке в Studio.

- **AI Gateway** (`ai/gateway.mjs`) — единственная точка, где content-api знает о конкретном провайдере (AI Generation Architecture, `docs/07`). Провайдер выбирается переменной окружения `AI_PROVIDER` (по умолчанию `template`).
- **Template Provider** (`ai/providers/template-provider.mjs`) — в этом деплое нет ключа доступа ни к одной внешней LLM, поэтому по умолчанию работает честный детерминированный генератор: валидный, проходящий гейт качества скелет дела (функция, тесты, брифинг-заготовка) под одну из шести тем, без единого сетевого вызова наружу. Это не имитация настоящего AI — это то же самое разделение ответственности (Gateway/Provider), просто с самым простым провайдером из возможных.
- Подключить настоящую LLM позже: добавить `ai/providers/<name>-provider.mjs` с той же сигнатурой `generate({ topic, existingIds })` и выставить `AI_PROVIDER=<name>` — ни Studio, ни остальной content-api трогать не придётся.
- `POST /api/generate-draft` ничего не сохраняет — черновик публикуется тем же `PUT /api/content/:id` и проходит тот же гейт качества, что и любая ручная правка (AI не имеет привилегированного пути записи).

## Analytics (v0.5)

Player шлёт копию каждого своего игрового события (`apps/player/js/engine.js: logGameEvent`) на сервер — асинхронно, best-effort: сбой сети никогда не блокирует игру и не влияет на локальный журнал `S.events` (Event Architecture, `docs/13`, раздел 15 — Analytics некритична).

- `services/content-api/data/events.jsonl` — неизменяемый (append-only) журнал, одна строка — одно событие.
- `GET /api/analytics` — агрегаты по журналу: сколько раз дело взято/закрыто, попытки проверки и их успешность, использование подсказок.
- Studio → «Аналитика →» — дашборд по данным реальных прохождений, а не по демо-числам: пока ни один агент не сыграл ни одного дела, экран честно об этом сообщает.

## Content API

`services/content-api/server.mjs` — весь бэкенд проекта на сегодня: без фреймворков, без БД, только встроенные модули Node (`http`, `fs/promises`).

- `GET /api/content` — весь стор Studio-правок
- `PUT /api/content/:id` — создать/обновить дело (частичный апдейт стора — конкурентные правки разных дел не затирают друг друга); отклоняется с `422` и списком ошибок, если не пройден гейт качества
- `DELETE /api/content/:id` — сбросить правку к сиду
- `POST /api/generate-draft` — черновик дела через AI Gateway (см. выше)
- `POST /api/events`, `GET /api/analytics` — журнал игровых событий и агрегаты (см. Analytics выше)
- Раздаёт всю остальную статику репозитория; блокирует `.git/.github/.claude/.gitignore` (эти каталоги не должны быть публичными, даже если репозиторий склонирован прямо в корень, который сервис раздаёт)

## Безопасность (обзор перед Beta)

Контент дел теперь приходит не только от разработчика, а и от методистов через Studio и от AI-провайдера — это меняет модель угроз: любой текст в деле нужно считать untrusted-вводом. При подготовке к Beta найдено и исправлено:

- Название дела вставлялось без экранирования на экране рангов терминала (`apps/player/js/screens.js`) — исправлено (`esc()`).
- `id` дела не проверялся на допустимые символы, а вставляется как HTML-атрибут (`data-case="${c.id}"`) в нескольких экранах Player — теперь `services/content-api/server.mjs` (и `scripts/check.mjs` для сида) отклоняет id, содержащий что-либо, кроме строчных латинских букв, цифр и дефиса.
- Остальные текстовые поля (брифинг, финал, версии, материалы, улики) уже были защищены — часть через `esc()`, часть через `textContent` (не исполняет разметку в принципе).
- Известный остаточный риск, не закрытый в этой итерации: `id` вложенных объектов (улики `evidence[].id`, материалы) не проверяются тем же правилом — при недобросовестном методисте это тот же класс уязвимости на менее заметных экранах. Studio — внутренний инструмент для доверенных пользователей, поэтому это принято как осознанный остаточный риск, а не блокер Beta.
- Исполнение решений игрока (`new Function` в браузере) происходит только в собственной вкладке игрока — не на сервере, не в общем процессе — поэтому не создаёт риска для других пользователей или инфраструктуры даже при неаккуратном решении.

## Резервное копирование

`scripts/backup.sh` — архивирует `services/content-api/data/` (Studio-правки + журнал событий, единственные данные проекта, не восстановимые из git) в `.tar.gz` с ротацией (по умолчанию хранится 14 копий). Настроено через cron на VPS-деплое (см. память сессии `vps-deployment`).

## Дорожная карта

Проект следует поэтапному плану (`docs/16-roadmap.md`): Foundation → Core Platform → MVP → v0.2 → v0.3 (Studio) → v0.4 (AI) → v0.5 (Analytics) → Beta → Release v1.0.

Текущий этап — **Beta**: минимум одно дело (`case-011`, «Особый список») прошло полный производственный цикл — сгенерировано AI Gateway, доработано в форме Studio, сыграно и проверено в Player, перенесено в постоянный сид; проведена и закрыта проверка безопасности; настроено резервное копирование.

Текущий этап — **v0.5, Analytics** поверх v0.4 (AI Generation, с оговоркой про шаблонный провайдер по умолчанию) и реального Publishing из v0.3.

## Проверка качества

```bash
node scripts/check.mjs
```

Проверяет синтаксис всех JS/MJS-файлов Player, Studio, общих пакетов и content-api, а также базовые инварианты игровых данных (у каждого играбельного дела есть задача, стартовый код, улики с тестами, подсказки, брифинг, финал и хотя бы одна верная версия отчёта). Та же логика инвариантов встроена в `services/content-api/server.mjs`. Прогоняется автоматически в CI на каждый push и pull request.
